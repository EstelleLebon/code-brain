## 🏛️ Décisions techniques

### Interface CognitiveExecutionRuntime comme seule dépendance d'AdaptiveCognitiveLoop
- **Choix** : `AdaptiveCognitiveLoop` reçoit uniquement `CognitiveExecutionRuntime` (interface) — jamais `ClusterTrustManager`, `ConsensusHealthMonitor`, etc. directement
- **Alternatives écartées** : injection directe des composants distribués (couplage fort), héritage de `DistributedCognitiveLoop` (violerait SRP)
- **Raison** : la stratégie cognitive (scoring, sélection de mode) doit pouvoir être testée avec un stub sans instancier l'infrastructure distribuée ; cela permet aussi de substituer un `LocalExecutionRuntime` pour les environnements mono-nœud

### Dégradation progressive automatique dans ClusterTrustManager
- **Choix** : après 10 incidents cumulés (failures + isolations + divergences) ou trustScore ≤ 0.1, un nœud est auto-quarantiné sans throw
- **Alternatives écartées** : seuil basé uniquement sur trustScore (trop sensible aux deltas variables), throw explicite (interdit par les contraintes du projet)
- **Raison** : garantir que le système reste toujours recoverable — la quarantaine est réversible via `releaseNode(nodeId, recoveryTrust)`

### Fingerprints déterministes sans crypto ni Math.random
- **Choix** : les fingerprints sont des chaînes construites par l'appelant (`"fp-{nodeId}-{round(score*100)}"` dans `DistributedExecutionRuntime.synchronizeCluster`) — le `SplitBrainDetector` compare des chaînes opaques
- **Alternatives écartées** : SHA-256 (interdit), UUID/nanoid (non déterministe)
- **Raison** : contrainte stricte — toutes les opérations doivent être replayables ; les fingerprints basés sur l'état logique (trustScore × 100 arrondi) sont stables pour les mêmes entrées

### Benchmarks dans la suite de tests unitaires
- **Choix** : les benchmarks sont dans `multi-node-benchmark.test.ts` avec des assertions sur les latences (< 50ms/cycle pour 1-3 nœuds, < 100ms pour 5 nœuds)
- **Alternatives écartées** : suite benchmark séparée (complique la CI), mesures sans assertions (pas de régression détectable)
- **Raison** : les budgets de latence sont vérifiables en CI ; les assertions `avgCycleMs < 50` détectent les régressions de performance
