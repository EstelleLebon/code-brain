## 🏛️ Décisions techniques

### LocalExecutionRuntime — trust décroissant déterministe sans randomness
- **Choix** : trust part de 1.0, décroit de 0.05/cycle en cas d'échec, remonte de 0.02 en cas de succès. Seuils fixes : `≥0.6 → success`, `≥0.4 → timeout`, `≥0.2 → failure`, `<0.2 → partition`.
- **Alternatives écartées** : table de lookup par itération (trop rigide), miroir du `DistributedExecutionRuntime` (ajouterait des dépendances cluster).
- **Raison** : simple, entièrement déterministe, replayable via `reset()`, compatible avec le contrat `CognitiveExecutionRuntime` sans importer `ClusterTrustManager`.

### persistDecision — appel privé depuis runCycle, pas méthode publique
- **Choix** : `persistDecision()` est private, appelée automatiquement dans `runCycle()`. `attachEventStore()` est la seule API publique.
- **Alternatives écartées** : méthode `persist()` publique à appeler manuellement (charge sur les appelants), persistance dans le constructeur (couplage fort).
- **Raison** : zéro friction pour l'appelant — si un store est attaché, tout est persisté transparentement. Sans store, aucun effet.

### getActiveNodeIds — fallback 'local' plutôt qu'erreur
- **Choix** : si `NodeRegistry.availableNodes()` retourne [], on utilise `['local']` comme nodeId fictif.
- **Raison** : `executeGoal()` est utilisé en mode mono-nœud sans cluster enregistré. Lever une erreur bloquerait tous les tests existants. Le fallback permet à `LocalExecutionRuntime` de fonctionner naturellement.

### checkConsistency / checkMemoryDivergence — event-driven, sans throw
- **Choix** : les deux méthodes restent privées, déclenchées via bus (`subscribe` dans le constructeur). Elles émettent des `TrustAlert` via `emitAlert()` sans jamais throw.
- **Raison** : cohérent avec le reste du `SplitBrainDetector` — toutes les détections sont passives et collectées, pas actives. Le bus découple l'émetteur du détecteur.

### Points d'attention
- `AdaptiveCognitiveLoop.attachEventStore()` doit être appelé **avant** le premier `runCycle()` sinon les premières décisions ne sont pas persistées (c'est le cas dans `CodeBrain` — attaché dans le constructeur).
- `LocalExecutionRuntime` ne simule pas de partitions réseau réelles — c'est intentionnel. Pour les tests de chaos, utiliser `DistributedExecutionRuntime` + `ChaosEngine`.
- `checkConsistency()` et `checkMemoryDivergence()` sont **privées** dans `SplitBrainDetector` — elles ne s'activent que via les événements bus (`consensus_resolved` / `memory_replicated`). Pour les déclencher dans les tests, publier ces événements sur le bus avec les bons champs typés.
- Les tests `v6.1-adaptive-runtime.test.ts` utilisent `DistributedEventBus` avec des payloads stricts : `{ type: 'consensus_resolved', proposalId: string, outcome: boolean }` et `{ type: 'memory_replicated', sourceNodeId: string, targetNodeId: string, memoryId: string }`.
