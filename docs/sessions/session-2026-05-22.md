## 🏛️ Décisions techniques

### Watcher integration opt-in
- **Choix** : `InvalidationEngine` et `TransactionCoordinator` passés en options optionnelles du constructeur — watcher fonctionne sans
- **Alternatives écartées** : toujours activer la transaction (aurait cassé les tests existants), sous-classer Watcher
- **Raison** : backward compatibility absolue ; les tests existants du watcher tournent sans modification

### Transactions via SQLite natif BEGIN/COMMIT/ROLLBACK
- **Choix** : `CognitiveTransaction` wrap directement les primitives SQLite de better-sqlite3
- **Alternatives écartées** : librairie externe de transaction, transactions purement in-memory
- **Raison** : better-sqlite3 est déjà présent, les transactions SQLite sont atomiques par définition, zéro dépendance supplémentaire

### Versioning via SELECT + INSERT OR REPLACE
- **Choix** : SELECT version existante → INSERT OR REPLACE avec version+1
- **Alternatives écartées** : trigger SQLite, colonne AUTOINCREMENT séparée
- **Raison** : contrôle explicite côté applicatif, pas de magie SQL, testable unitairement

### RetrievalValidator comme post-processing non-bloquant
- **Choix** : validation après retrieval, résultat émis en telemetry uniquement — ne filtre pas les résultats
- **Alternatives écartées** : filtrer les chunks invalides, bloquer le retour si validation échoue
- **Raison** : la spec dit "préférer réduire le contexte plutôt que retourner du faux" — implémenté en deux étapes : reporter d'abord, filtrer en P2

### Semantic IR foundations-only
- **Choix** : uniquement interfaces + SemanticIR POJO sans intégration dans le pipeline
- **Alternatives écartées** : intégrer dans le watcher ou l'API, implémenter une opération réelle
- **Raison** : spec explicite "NE PAS implémenter un IR complet" — le but est de poser les types pour les futures transformations déterministes
