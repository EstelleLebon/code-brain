## 🏛️ Décisions techniques

### BaseRunner : execFile async, pas de shell
- **Choix** : `execFile` (promisifié) — pas de shell intermédiaire
- **Alternatives écartées** : `exec` avec shell=true (injection de commandes possible), `spawn` (verbeux pour capturer stdout/stderr)
- **Raison** : sécurité + cohérence avec le pattern existant dans la codebase

### RuntimeValidationPipeline : fail-fast sur erreurs non-optional
- **Choix** : arrêt au premier stage `failure` sauf si `optional: true`
- **Alternatives écartées** : toujours continuer (comme ValidationPipeline v1) ; toujours arrêter
- **Raison** : un build cassé rend les résultats de tests non-fiables

### OutcomeClassifier : hiérarchie build > typecheck > test > warning
- **Choix** : `build+test failure = failure`, `build seul = regression`, `test seul = regression`, `warning seul = partial_success`
- **Raison** : build/typecheck sont des prérequis aux tests — leur échec seul indique une régression structurelle

### FailureMemory : déduplication par similarity score (seuil 0.6)
- **Choix** : `overlapScore = |intersection| / max(|A|, |B|)` sur `structuralContext[]`
- **Alternatives écartées** : égalité exacte (trop strict), embeddings (trop lourd)
- **Raison** : deux patterns avec 60%+ de contexte commun représentent le même problème avec variations mineures

### scoreToLevel exportée depuis RiskAssessmentEngine
- **Choix** : `export function scoreToLevel()` dans `RiskAssessmentEngine.ts`
- **Raison** : `RuntimeAwareRiskAssessor` en a besoin pour convertir son score ajusté — évite la duplication
