## 🏛️ Décisions techniques

### makeDb() dans les tests : CREATE_TABLES_SQL + MIGRATIONS

- **Choix** : `db.exec(CREATE_TABLES_SQL)` puis boucle sur `MIGRATIONS` (v2, v3, v4)
- **Alternatives écartées** : copier les DDL dans chaque fichier de test ; instancier la vraie classe `DB`
- **Raison** : les migrations v2/v3 font des `ALTER TABLE` sur des tables qui doivent exister via `CREATE_TABLES_SQL` — sans ce prérequis, les migrations échouent silencieusement et les stores ne fonctionnent pas

### filterResults : nouvelle collection, pas de mutation

- **Choix** : `filterResults()` construit un nouveau tableau `filtered[]`, n'altère jamais `results`
- **Alternatives écartées** : `results.splice()`, filtre in-place
- **Raison** : déterminisme garanti, lisibilité, testabilité — les résultats originaux restent accessibles si besoin de diagnostic

### isTruthLevelAtLeast : `current <= required`

- **Choix** : ordinal numérique (STRUCTURAL=0 = meilleur) → `current <= required` = "au moins aussi bon que"
- **Alternatives écartées** : comparaison inverse, table de lookup
- **Raison** : l'enum est ordonné numériquement, l'expression `<=` est lisible et directement mappée sur la sémantique "meilleur ou égal"

### MemoryCompactor : VACUUM non-obligatoire dans les tests

- **Choix** : `compact({ vacuum: false })` dans les tests
- **Raison** : WAL mode + `:memory:` DB — `VACUUM` sur une DB in-memory en WAL provoque des erreurs dans certaines versions de better-sqlite3

### CalibrationStore : INSERT OR IGNORE (pas upsert)

- **Choix** : `INSERT OR IGNORE` pour les calibration records — chaque record est immutable et timestampé
- **Alternatives écartées** : `ON CONFLICT DO UPDATE` (comme FailureMemoryStore)
- **Raison** : un record de calibration est une observation historique, pas un état mutable — on ne doit jamais écraser un record passé
