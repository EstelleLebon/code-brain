## 🏛️ Décisions techniques

### ASTMutation : positions de caractères, pas de lignes
- **Choix** : `startIndex` / `endIndex` en offset de caractère dans la string source (comme tree-sitter)
- **Alternatives écartées** : numéros de ligne + colonnes (ambiguïtés avec tabs/unicode), string matching (fragile)
- **Raison** : tree-sitter expose exactement `node.startIndex` / `node.endIndex` — zéro conversion, zéro ambiguïté

### Application des mutations en reverse order
- **Choix** : trier par `startIndex` décroissant avant d'appliquer
- **Alternatives écartées** : recalculer les offsets après chaque mutation, reconstruire l'AST entre mutations
- **Raison** : les mutations en reverse order préservent les positions des mutations non encore appliquées — algorithme O(n log n), déterministe, sans état mutable entre mutations

### TransformationEngine : tree-sitter avec fallback regex
- **Choix** : même pattern que `src/parser/ast.ts` — try/require tree-sitter, regex si indisponible
- **Alternatives écartées** : tree-sitter obligatoire, recast/babel pour la transformation
- **Raison** : cohérence avec la base de code existante ; les tests passent dans les deux environnements

### ValidationPipeline : checks non-bloquants par défaut
- **Choix** : tous les checks continuent même si un précédent échoue ; `riskScore` s'accumule
- **Alternatives écartées** : fail-fast au premier check en erreur
- **Raison** : permet de collecter tous les problèmes en un seul passage, utile pour l'affichage à l'utilisateur avant correction

### SemanticExecutor : dry-run toujours success=true
- **Choix** : en `dryRun=true`, `success` est toujours `true` indépendamment du trust/validation
- **Alternatives écartées** : appliquer quand même les politiques en dry-run
- **Raison** : le dry-run doit toujours retourner le plan et les diffs pour que l'utilisateur puisse inspecter — bloquer en dry-run rendrait l'exploration impossible

### Trust policies séparées du risk engine
- **Choix** : `TrustEvaluator` reçoit une `TrustPolicy` à la construction, découplé de `RiskAssessmentEngine`
- **Alternatives écartées** : risk + trust dans le même composant, policies hardcodées
- **Raison** : permet d'injecter des policies différentes par contexte (CI strict vs dev local), testable indépendamment

### Points d'attention
- **`extract_interface` ne fonctionne qu'avec tree-sitter** — sans tree-sitter, `TransformationEngine` retourne `[]` pour cette opération (regex fallback non implémenté pour l'extraction d'interface)
- **`move_function` avec fichier destination** — la mutation d'insertion dans le fichier destination est générée seulement si `ctx.allFiles` contient déjà ce fichier ; si le fichier destination n'existe pas dans la map, la mutation d'insertion est silencieusement ignorée
- **`inject_dependency` et `split_module`** — enregistrés dans `OperationRegistry` (via `TransformationEngine`) mais `planMutations` retourne `[]` pour ces types — ce sont des stubs
- **dry-run est le mode par défaut** — `createExecutionContext(..., true)` ; passer `false` pour écrire sur disque
- **Trust `PERMISSIVE_TRUST_POLICY`** — auto-approve jusqu'à `medium` risk ; à utiliser dans les tests pour éviter d'être bloqué par la policy par défaut (`low` seulement)
- **SemanticReplayLog et ReplayLog v1.6** — deux logs distincts, tous deux in-memory, tous deux perdus au redémarrage
