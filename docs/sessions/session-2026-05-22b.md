## 🏛️ Décisions techniques

### Isolation des erreurs dans executeAsync()
- **Choix** : chaque étape post-exécution (feedbackLoop, selfHealingEngine, metricsAggregator) est wrappée dans un `try/catch` silencieux
- **Alternatives écartées** : propager les erreurs de la boucle cognitive vers l'appelant
- **Raison** : les erreurs de learning/healing ne doivent jamais casser l'exécution principale — principe fondamental de la v3.5

### AdaptiveTrustPolicy — seuils d'adaptation (3 failures / 5 successes)
- **Choix** : `recentFailures >= 3` → tighten, `recentSuccesses >= 5 && recentFailures === 0` → loosen
- **Alternatives écartées** : seuils configurables, algorithme de Bayesian update
- **Raison** : seuils simples et déterministes, testables directement, ajustables via `windowSize`

### RollbackHeuristics — ciblage causal via metadata des signaux
- **Choix** : extraire `signal.metadata.files` pour identifier les artifacts causaux → rollback `targeted` si subset < total
- **Alternatives écartées** : toujours rollback complet sur failure
- **Raison** : rollback partiel préserve le travail correct sur les fichiers non impactés

### WorkingMemory.isExpired() — `>=` vs `>`
- **Choix** : `Date.now() - lastActivity >= this.ttlMs`
- **Raison** : permet les tests avec `ttl=0` sans race condition (le test `isExpired returns true for ttl=0` échouait avec `>`)

### ModeSelector — priorité HOTFIX absolue
- **Choix** : `isHotfix` court-circuite toutes les autres règles avant même de vérifier les failures
- **Raison** : un hotfix critique ne doit jamais être dégradé en RECOVERY ou EXPLORATION même si le contexte est instable
